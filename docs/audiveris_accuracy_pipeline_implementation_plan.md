# Audiveris精度向上パイプライン 実装プラン（詳細版）

対象: **Audiveris完全自動・人手修正なし**

更新日: 2026-04-24

---

## 0. このドキュメントの狙い

この計画は、単なるTo-Do列挙ではなく、以下を満たすための**実装可能な設計書**です。

- 何を先に作るべきか（優先度）
- 何をもって完了とするか（DoD: Definition of Done）
- どのメトリクスで改善を判定するか（評価軸）
- 失敗時にどこへ戻るか（再試行・分岐ルール）
- どこに時間/計算資源がかかるか（コスト管理）

---

## 1. 全体方針（重要）

### 1.1 優先順位（効き × 依存 × 工数）

1. **基盤設計（再現性・比較可能性）**
2. **五線基準の前処理 + 品質ゲート**
3. **マルチ試行 + 自動選択**
4. **拍数補正（リズム成立）**
5. **和音/voice再構成（演奏可能性）**
6. 音高/構造補正
7. フィードバックループで継続改善

### 1.2 成果物の単位

- ページ単位成果物（`page_result`）
- システム単位成果物（`system_result`）
- スコア全体成果物（`score_result`）

### 1.3 まず守る原則

- **Fail Fast**: 品質ゲート不合格を後段へ流さない
- **Small Fix First**: 後処理は最小変更
- **Trace Everything**: すべての判断にログを残す

## 2. KPI設計（改善判定の軸）

## 2.1 オンラインKPI（本番監視）

- `valid_xml_rate`: 破損なしXML比率
- `bar_rhythm_valid_rate`: 拍整合小節率
- `pitch_out_of_range_rate`: 音域外音率
- `reprocess_rate`: 再処理発生率
- `drop_rate`: 破棄率
- `mean_total_score`: 最終採択スコア平均

## 2.2 オフラインKPI（開発評価）

- `note_f1`（参照譜がある場合）
- `onset_f1`
- `duration_mae`
- `chord_grouping_f1`
- `voice_consistency_score`

## 2.3 目標値（初期）

- valid_xml_rate: **95%+**
- bar_rhythm_valid_rate: **98%+**
- drop_rate: **10%未満**
- reprocess_rate: **30%未満**

> 目標値はデータセット特性に応じて四半期ごとに再設定する。

## 3. データセット戦略（評価の土台）

## 3.1 セット構成

- `golden_set`（100〜300ページ）: 人手確認済み参照あり
- `hard_set`（50〜100ページ）: 低品質スキャン/古典印刷/密譜
- `smoke_set`（20ページ）: 回帰確認用（CI向け）

## 3.2 分割

- Train（パラメータ探索）: 60%
- Dev（閾値調整）: 20%
- Test（固定）: 20%

## 3.3 バージョン運用

- `dataset_version` を必ず記録
- 評価結果は `param_set_id + dataset_version` で保存

## 5. Phase 0: 基盤設計（最優先）

## 5.1 実装タスク

### P0-1. ステージ分離
- `run_preprocess(page, params)`
- `run_omr(page, params)`
- `run_postprocess(xml, params)`

### P0-2. ジョブ管理
- ページ単位キュー投入
- 最大並列数 `N` を設定可能
- タイムアウト（例: 180秒/ページ）
- リトライ（例: 最大2回、指数バックオフ）

### P0-3. ログ・監査
- 構造化ログ（JSON Lines）
- `job_id`, `image_id`, `stage`, `error_code` を必須化
- 失敗時に入力/出力パスを記録

### P0-4. パラメータ管理
- YAML + JSON Schemaバリデーション
- `param_set_id` をセマンティックバージョンで管理

## 5.2 DoD

- 同一入力・同一paramで再実行し、同一スコア±許容誤差で再現
- エラーの90%以上がログのみで原因分類可能

## 6. Phase 1: 前処理（五線中心）

## 6.1 入力正規化

### 実装
- PDFレンダリングDPIを2条件（300/400）で比較
- グレースケール変換
- CLAHEまたは局所コントラスト補強
- 余白トリミング（外接矩形 + マージン固定）

### 記録メトリクス
- `dpi`
- `fg_ratio`（前景画素比）
- `margin_px`

## 6.2 五線ベース正規化（最重要）

### 実装
- staff space推定（行方向投影 + ピーク間隔）
- 目標20〜25pxにスケーリング
- 行単位Y整列（局所回帰で補正）
- 五線本数異常フラグ（5本でなければ警告）

### 閾値案
- `staff_space_px` が [14, 35] 外なら低信頼
- `staff_line_count_error_rate > 0.15` で再処理候補

## 6.3 幾何補正

- Houghで全体傾き推定
- 行単位warp（薄い歪み補正）
- 写真入力のみ湾曲補正を有効化

## 6.4 二値化・ノイズ制御

- Sauvola（window, k を探索）
- 背景推定→減算→再二値化
- 小物体除去（面積 + 円形度）
- 軽いクロージング（核サイズを最小）

## 6.5 品質ゲート（必須）

### スコア定義（0〜1）
- `staff_detect_score`
- `noise_score`
- `skew_score`
- `symbol_connectivity_score`

総合: `q = 0.4*staff + 0.2*noise + 0.2*skew + 0.2*connectivity`

### 分岐
- `q >= 0.75`: 通過
- `0.55 <= q < 0.75`: パラメータ切替で再処理
- `q < 0.55`: 破棄（ログ保存）

## 6.6 DoD

- quality gate導入後、壊れXML率をベースラインから20%以上改善

---

## 7. Phase 2: 認識制御（Audiveris）

## 7.1 Audiveris設定最適化

- OCR関連（歌詞/コード/指番号）を無効化
- 装飾記号クラス抑制
- 不要クラス削減

## 7.2 マルチ試行設計

### trial生成
- `trial_a`: 低ノイズ強調
- `trial_b`: ステム保護優先
- `trial_c`: 薄譜面向け高感度

### 実行
- 並列実行（CPU予算内）
- 各trialで中間ログ取得

## 7.3 ステップ監視

- GRID段階で五線本数チェック
- staff間隔分散チェック
- 異常なら即abortして別trialへ

## 7.4 結果フィルタ

- 小節数ゼロを除外
- 異常音符密度（例: ページ平均±3σ外）を除外
- XML構文不正を除外

## 7.5 部分再処理（第2段階導入）

- 異常小節を局所再切り出し
- 小節単位で再OMR
- 元結果と差分統合

## 7.6 DoD

- 単一試行比で最終採択スコアが有意改善
- 全損ページ率を30%以上削減

---

## 8. Phase 3: 後処理（楽譜成立性）

## 8.1 リズム補正（最重要）

### 処理順
1. 小節拍数計算
2. 不足は休符候補を最小挿入
3. 過剰は誤検出音の削除候補を探索
4. 削除で悪化する場合は和音化/音価再配分

### ルール
- 変更コスト関数 `C = w1*insert + w2*delete + w3*duration_edit`
- C最小案を採択

## 8.2 タイミング補正

- onsetを格子へ量子化
- 近接onsetを同時刻クラスタ化

## 8.3 音高補正

- K-S法で調性推定
- スケール外音に低コスト補正候補を割当
- n-gramで不自然遷移を抑制
- オクターブ誤りを近傍文脈で補正

## 8.4 音域・密度フィルタ

- 楽器音域外ノート除外
- 局所密度スパイク検出
- 孤立短音（低信頼）を削除候補に

## 8.5 和音/voice再構築

- 同時刻ノート統合（和音）
- 横ズレ許容窓で再束ね
- 右手/左手voice推定
- 単発ノイズvoiceを削除

## 8.6 記号整理

- タイ整合
- スラーは削除または無視
- 装飾音/ダイナミクス除去

## 8.7 構造補正

- 小節数整合
- リピート/D.C./D.S. は削除か展開で統一

## 8.8 DoD

- bar_rhythm_valid_rate 98%以上
- MIDI再生不能率を実用閾値以下へ

## 9. Phase 4: 評価・選択

## 9.1 スコア関数

`total = 0.35*rhythm + 0.20*range + 0.15*density + 0.15*tonality + 0.15*structure`

## 9.2 選択ロジック

- 候補ごとに`total`算出
- 最大値を採択
- `total < theta` で再試行or破棄

## 9.3 閾値チューニング

- DevセットでROC相当の解析
- 破棄率と品質を見ながら `theta` 決定

## 9.4 DoD

- 手動選択との差分が許容範囲
- 再試行コストが運用上限内

## 15. 最終結論

最短で成果を出す順序は以下で固定する。

1. **再現可能な基盤**を先に作る
2. 五線中心前処理と品質ゲートで入力品質を底上げ
3. マルチ試行 + 自動選択でAudiverisの不安定性を吸収
4. リズム補正とvoice再構築で「演奏可能な譜面」に寄せる
5. 失敗原因を学習して次のパラメータへ反映

この5段を守ることで、精度改善を「単発」ではなく「継続的な改善ループ」にできる。
